## Overview

The Power Debug Agent is an AI-powered tool for analyzing and comparing power reports from IC design verification runs. It uses MCP (Model Context Protocol) servers to provide comprehensive power analysis, annotation quality assessment, and automated comparison capabilities.

## Key Capabilities

- **Rollup Report Analysis**: Parse and analyze power rollup reports (`.rollup.rpt` files)
- **Cell Activity Analysis**: Parse detailed cell-level activity reports (`.all.activity.rpt` files)
- **Cell Power CSV Analysis**: Parse cell-level power data by VT class, CIM type, and cell family
- **Comparative Analysis**: Compare power metrics between reference and experiment runs
- **Annotation Quality**: Assess activity annotation quality and coverage
- **Design Validation**: Verify design consistency across runs
- **VT Distribution Analysis**: Compare threshold voltage distributions and their power impact
- **Automated Insights**: Generate actionable recommendations based on findings
- **Mapping Report Analysis**: Analyze FSDB-to-design signal mapping quality
- **Empower vs PrimePower Comparison**: Compare Empower runs against PrimePower using the shared rollup and activity report formats (see [Empower Runs](#empower-runs) section)
- **Timing Margin Analysis**: Restore PrimeTime sessions and analyze per-pin slack, Vt-class breakdown of timing-clean cells, and clock tree latency to identify power reduction candidates (see [Timing Tools](#timing-tools))
- **Spatial Power Analysis**: Read primePD spatial reports to identify power hotspot pixels on the die, drill into per-cell power within hotspots, and look up cell placement coordinates (see [Spatial Tools](#spatial-tools))

### Empower Run Directory Structure

Empower runs produce a **subset** of the reports generated by PrimePower. Only the rollup and activity reports are guaranteed to be present:

```
<empower_run_dir>/
   └── reports/               # Empower run reports (subset of PrimePower)
       ├── <design>.rollup.rpt.gz           ✅ Present
       ├── <design>.all.activity.rpt.gz     ✅ Present
       ├── <design>.compressed_cell_power.csv.gz  ❌ Not available
       ├── <design>.all_inclusive_mapping.rpt.gz  ❌ Not available
       ├── <design>.all_inclusive_mapping_summary.rpt.gz  ❌ Not available
       └── <design>_variables.rpt.gz        ❌ Not available
```

When comparing an Empower run against a PrimePower run, only tools that rely solely on the rollup and/or activity reports can be used. See the [Empower Runs](#empower-runs) section for the compatible tool list and recommended workflow.

### Discovery & Setup Tools

#### 0. `discover_reports_for_comparison` ⭐ **RECOMMENDED FIRST STEP**
**Purpose**: Discover and return all report directory paths and key files for both runs upfront

**Parameters**:
- `ref_power_extraction` (string): Path to reference power_extraction directory
- `exp_power_extraction` (string): Path to experiment power_extraction directory
- `partition_name` (string): Partition name (e.g., "par_ooo_int")
- `test_name` (string): Test name (e.g., "ipc_1_5_dsb")

**Returns**: Dictionary with:
- `ref_reports_dir` / `exp_reports_dir`: Report directory paths
- `ref_rollup_path` / `exp_rollup_path`: Rollup report paths
- `ref_cell_power_csv` / `exp_cell_power_csv`: Cell power CSV paths
- `ref_variables_path` / `exp_variables_path`: Variables report paths
- `ref_activity_path` / `exp_activity_path`: Activity report paths
- `ref_mapping_summary` / `exp_mapping_summary`: Mapping report paths
- `files_found`: Boolean flags for each file type

**Example Usage**:
```python
# Call this FIRST to discover all files
paths = discover_reports_for_comparison(
    "/path/to/ref/power_extraction",
    "/path/to/exp/power_extraction",
    "par_ooo_int",
    "ipc_1_5_dsb"
)

# Then use the returned paths directly:
ref_metrics = read_rollup_report(paths['ref_rollup_path'])
exp_metrics = read_rollup_report(paths['exp_rollup_path'])
ref_cell_data = read_cell_power_csv(paths['ref_cell_power_csv'])
```

**Benefits**:
- ✅ All file paths discovered in one operation
- ✅ No repeated directory searching
- ✅ Clear visibility of which files exist
- ✅ Eliminates path construction errors

### Core Analysis Tools

#### 1. `read_rollup_report`
**Purpose**: Parse a power rollup report file (supports `.gz` compression)

**Parameters**:
- `file_path` (string): Absolute path to rollup report file

**Returns**: Dictionary of all metrics (name → value)

**Example Usage**:
```python
metrics = read_rollup_report("/path/to/design.rollup.rpt.gz")
print(f"Total Power: {metrics['TotPwr']}")
print(f"AQM: {metrics['AQM']}")
```

**Key Metrics Returned**:
- Power: `TotPwr`, `DynPwr`, `LkgPwr`, `DynCellPwr`, `BBoxPwr`, `GlobalClkPwr`, `SeqPwr`, `CombPwr`
- Activity: `AvgTRClk`, `AvgTRData`, `DataSumAF`, `ClkSumAF`
- Annotation: `AQM`, `InpAnnotated`, `IntAnnotated`, `BBoxAnnotated`, `AvgVal`
- Design: `CellCount`, `SeqCount`, `SeqBitCount`, `Ctot`, `TotalZ`
- Config: `Voltage`, `Temperature`, `TEST`, `RTL_FSDB`, `FSDB_Start`, `FSDB_End`

---

#### 2. `read_cell_power_csv`
**Purpose**: Parse compressed_cell_power.csv.gz file for detailed cell-level power analysis

**Parameters**:
- `csv_path` (string): Absolute path to compressed_cell_power.csv.gz file

**Returns**: Dictionary with comprehensive power breakdown:
- `totals`: Overall totals (leakage_uW, dynamic_uW, total_uW, cell_count)
- `by_vt_class`: Power breakdown by VT class (ULVT, LVT, ULVTLL, etc.)
- `by_cim_type`: Power breakdown by CIM type (Buffer, Inverter, Complex, Nand, etc.)
- `by_cell_family`: Power breakdown by cell family
- `top_templates_by_leakage`: Top-20 leaking cell templates

**Example Usage**:
```python
cell_data = read_cell_power_csv("/path/to/design.compressed_cell_power.csv.gz")
print(f"Total Power: {cell_data['totals']['total_uW']:.2f} uW")
print(f"ULVT Leakage: {cell_data['by_vt_class']['ch_ulvt_mac_3n']['leakage_uW']:.2f} uW")
```

**VT Classes Reported**:
- `ch_ulvt_mac_3n`: Ultra-low VT (fastest, highest leakage)
- `ch_lvt_mac_3n`: Low VT (balanced speed/power)
- `ch_ulvtll_mac_3n`: Ultra-low VT low leakage variant

---

#### 3. `read_cell_report`
**Purpose**: Parse cell-level activity report with pin-level details

**Parameters**:
- `file_path` (string): Absolute path to activity report (`.all.activity.rpt` or `.gz`)

**Returns**: CircuitData object with:
- `instances`: Dictionary of CellInstance objects (instance_name → cell_data)
- `instances_by_cim_type`: Grouping by CIM type
- `drivers_by_net`: Net → list of driver pins
- `sinks_by_net`: Net → list of sink pins

**Cell Instance Data**:
- `instance`: Instance name
- `template`: Cell template
- `library`: Library name
- `cim_type`: CIM type classification
- `specific_type`: Specific cell type
- `power`: Power breakdown (internal, switching, leakage)
- `pins`: List of pin records with:
  - `pin`: Pin name
  - `direction`: input/output/inout
  - `net`: Connected net name
  - `toggle_rate`: Toggle rate (TR)
  - `static_prob`: Static probability (SP)
  - `glitch_rate`: Glitch rate (TG)
  - `fanin`/`fanout`: Fan-in/fan-out counts
  - `src`: Annotation source ('file', 'prop', 'default')

### Timing Tools

These tools restore PrimeTime sessions and extract timing data for power-relevant analysis. They auto-detect the correct PrimeTime version from the session README and handle license configuration internally.

#### 22. `report_pin_margins`
**Purpose**: Report worst timing slack on output pins of all hierarchical cells

**Parameters**:
- `session_path` (string): Absolute path to a PrimeTime session directory (e.g. `/path/to/<partition>.pt_session.func.max_nom.T_85.typical`)
- `top_n` (int): Number of worst-slack pins to return (default 20). Set to 0 for all pins.

**Returns**: Multi-line report containing:
- PrimeTime version and session path
- Total output pins queried and worst slack value
- Table of pin names and max_slack values, sorted ascending (worst first)
- Count of pins with undefined slack

**Use when**:
- You need per-pin worst-slack data to identify timing-critical paths
- Correlating timing margins with spatial power hotspots — pins with large positive slack are Vt swap candidates
- You want a ranked list of the tightest pins in the design

---

#### 23. `analyze_vt_slack_margin`
**Purpose**: Classify cells by Vt class and identify those with sufficient timing margin for power-saving Vt swaps

**Parameters**:
- `session_path` (string): Absolute path to the PrimeTime session directory
- `margin_ps` (float): Slack threshold in ps (default 40.0). Pins with max_slack below this are considered tight.

**Returns**: Multi-line report containing:
- Cell summary: total, passing, failing counts with percentages
- Vt breakdown table of passing cells (ULVT, ULVTLL, LVT, SVT, HVT, DECAP, Other) with count and percentage columns
- Interpretation section highlighting highest-leakage Vt class with passing cells as swap candidates

**Use when**:
- Identifying power reduction candidates — the primary tool for finding cells with comfortable timing margin that could be Vt-swapped to reduce leakage
- You need a partition-wide Vt breakdown with pass/fail classification
- Correlating with spatial hotspot analysis — after `find_hot_pixels` and `get_cells_in_pixel` identify hot cells, use this to check if those cells have enough slack for Vt swap

---

#### 24. `report_clock_latency`
**Purpose**: Report clock tree latency to every sequential cell's clock pin

**Parameters**:
- `session_path` (string): Absolute path to the PrimeTime session directory
- `top_n` (int): Number of results to return (default 50). Set to 0 for all.
- `instance_pattern` (string): Optional glob pattern to filter cell instance names (e.g. "exe_vec/*"). Leave empty for all sequential cells.
- `sort_by` (string): Sort order — "latency" (longest first, default), "latency_asc" (shortest first), or "name" (alphabetical)

**Returns**: Multi-line report containing:
- Summary statistics: total clock pins, min/max/avg latency in ps and ns, clock skew (max − min)
- Table with columns: Cell Instance, Clk Pin, Latency (ps)

**Use when**:
- Identifying clock tree depth outliers — cells with longest clock paths often contribute to power hotspots
- Analyzing clock skew across the design
- Correlating with spatial power analysis — after `find_hot_pixels` reveals a hotspot, check whether deep clock trees contribute
- Scoping analysis to a specific hierarchy with `instance_pattern`

### Spatial Tools

These tools read primePD spatial reports from power extraction runs to analyze power density distribution across the die. The primePD directory is a sibling of the reports directory inside a power extraction run.

**Directory Structure**:
```
<partition>.<corner>.<test>/
├── reports/           # Power reports (rollup, activity, etc.)
└── primePD/           # Spatial power data
    ├── <partition>.xyv[.gz]                    # Cell placement coordinates
    └── flat_power_map/
        ├── <partition>.pixel_power_density.report   # Per-pixel power
        └── <partition>.cell_power_density.report    # Per-cell power with pixel assignment
```

#### 25. `find_hot_pixels`
**Purpose**: Identify the hottest pixels (highest power density) on the die

**Parameters**:
- `primepd_path` (string): Path to the primePD directory (or sibling reports directory — auto-resolved)
- `top_n` (int): Number of top pixels to return (default 10)
- `sort_by` (string): Sort key — 'tot_pwr', 'dyn_pwr', or 'lkg_pwr' (default 'tot_pwr')

**Returns**: Dictionary containing:
- `partition`: Partition name
- `summary`: Metadata (total_cells, total_power, etc.)
- `total_pixels`: Number of pixels in the report
- `hot_pixels`: Top N pixels sorted descending, each with pixel_id, row, col, y_nm, x_nm, dyn_pwr, lkg_pwr, tot_pwr

**Use when**:
- Starting hotspot analysis — call this first to find the highest-power spatial regions
- Comparing dynamic vs leakage hotspots by sorting by `dyn_pwr` or `lkg_pwr`
- You need a quick summary of the worst power density regions

---

#### 26. `get_cells_in_pixel`
**Purpose**: Get the top power-consuming cells in a specific spatial pixel

**Parameters**:
- `primepd_path` (string): Path to the primePD directory (or sibling reports directory)
- `pixel_id` (string): Pixel identifier from `find_hot_pixels`, e.g. "R_cell_5_5"
- `top_n` (int): Number of top cells to return (default 50)

**Returns**: Dictionary containing:
- `partition`: Partition name
- `pixel_id`: The queried pixel
- `cell_count`: Total cells in this pixel
- `total_pixel_power`: Sum of power for all cells in this pixel (Watts)
- `top_cells`: Top N cells sorted by total_power descending, each with instance_name, cell_template, cell_type, total_power, pct_power_in_pixel

**Use when**:
- After `find_hot_pixels` identifies hot regions, drill into each pixel to classify power-driving cells (clock tree, sequentials, buffers, etc.)
- Root-cause analysis: determine whether a hotspot is driven by clock cells, dense ULVTLL sequentials, large-drive buffers, etc.

---

#### 27. `read_xyv_cell_locations`
**Purpose**: Read physical x,y placement coordinates for cell instances from the XYV file

**Parameters**:
- `primepd_path` (string): Path to the primePD directory (or sibling reports directory)
- `instance_pattern` (string, optional): Regex pattern to filter instance names (case-insensitive), e.g. "cts_ccl_inv"
- `max_results` (int): Maximum cells to return (default 1000)

**Returns**: Dictionary containing:
- `partition`: Partition name
- `bounding_box`: Partition bounding box (x_um, y_um, x_max_nm, y_max_nm)
- `cells`: List of matching cells, each with instance_name, cell_template, x_um, y_um, bbox_nm

**Use when**:
- You need physical x,y locations of specific cell instances on the die
- Looking up coordinates for cells identified in hotspot analysis to understand spatial clustering
- Cross-referencing cell placement with power density data

### Workflow 0: Recommended Analysis Pattern (Most Efficient)

```python
# STEP 1: Discover all file paths upfront
paths = discover_reports_for_comparison(
    ref_power_extraction="/path/to/ref/power_extraction",
    exp_power_extraction="/path/to/exp/power_extraction",
    partition_name="par_ooo_int",
    test_name="ipc_1_5_dsb"
)

# Check what files were found
print(f"Files found: {paths['files_found']}")

# STEP 2: Use the discovered paths for analysis
ref_metrics = read_rollup_report(paths['ref_rollup_path'])
exp_metrics = read_rollup_report(paths['exp_rollup_path'])

# STEP 3: Perform specific analyses as needed
if paths['ref_cell_power_csv'] and paths['exp_cell_power_csv']:
    ref_cell_data = read_cell_power_csv(paths['ref_cell_power_csv'])
    exp_cell_data = read_cell_power_csv(paths['exp_cell_power_csv'])
    # Analyze VT distribution, etc.

# STEP 4: Generate automated conclusions
conclusions = generate_power_conclusions(
    paths['ref_rollup_path'],
    paths['exp_rollup_path']
)
```

## Empower Runs

### Overview

Empower is an alternative power analysis flow that produces the same `.rollup.rpt` and `.all.activity.rpt` report formats as PrimePower but does **not** generate the full suite of supplementary reports. This makes Empower runs directly comparable with PrimePower for top-level power metrics and activity analysis.

**Reports available in Empower:**
| Report | Available | Used by Tools |
|--------|-----------|---------------|
| `<design>.rollup.rpt[.gz]` | ✅ Yes | `read_rollup_report`, `compare_rollup_reports`, `analyze_power_breakdown`, `generate_power_conclusions` |
| `<design>.all.activity.rpt[.gz]` | ✅ Yes | `read_cell_report`, `find_activity_reports`, `compare_sequential_toggle_rates`, `analyze_multibit_sequentials` |
| `<design>.compressed_cell_power.csv.gz` | ❌ No | `read_cell_power_csv`, `compare_cell_family_power_for_test` |
| `<design>_variables.rpt[.gz]` | ❌ No | `read_variables_report`, `compare_variables_reports` |
| `<design>.all_inclusive_mapping_summary.rpt[.gz]` | ❌ No | `read_mapping_summary`, `find_mapping_reports` |

### Compatible Tools for Empower vs PrimePower Comparison

When one or both runs are from Empower, restrict analysis to these tools:

**Rollup-based (both Empower and PrimePower):**
- `read_rollup_report` — parse top-level power metrics from either run
- `compare_rollup_reports` — diff all rollup metrics between Empower and PrimePower
- `analyze_power_breakdown` — dynamic vs leakage waterfall comparison
- `generate_power_conclusions` — automated root-cause conclusions

**Activity-based (both Empower and PrimePower):**
- `read_cell_report` — parse pin-level activity detail
- `find_activity_reports` — locate rollup and activity files
- `compare_sequential_toggle_rates` — compare toggle rates on sequential output pins
- `analyze_multibit_sequentials` — compare multi-bit FF utilization and toggle statistics
- `report_annotation_rates` / `print_annotation_table` — evaluate annotation coverage

**Do NOT use for Empower runs:**
- `read_cell_power_csv` — requires `compressed_cell_power.csv.gz` (PrimePower only)
- `compare_cell_family_power_for_test` — requires `compressed_cell_power.csv.gz`
- `read_variables_report` / `compare_variables_reports` — requires `_variables.rpt`
- `read_mapping_summary` / `find_mapping_reports` — requires mapping reports
- `comprehensive_power_comparison` — attempts to use all report types; use the individual rollup/activity tools instead

### Important Notes for Empower Comparisons

- **AQM may differ** between Empower and PrimePower due to different annotation engines. Verify AQM from both rollup reports before drawing conclusions from dynamic power differences.
- **No VT/cell-family breakdown** is possible without `compressed_cell_power.csv.gz`. Leakage differences can only be analysed at the total level from the rollup.
- **No variable/configuration diff** is possible without `_variables.rpt`. Check rollup headers (Voltage, Temperature, FSDB window) manually for configuration consistency.
- **`discover_reports_for_comparison` may not apply** if the Empower run does not follow the standard `power_extraction/<partition>.<corner>.<test>/reports/` directory convention. In that case, pass rollup and activity paths directly to the individual tools.

## Report Interpretation Guide

### Power Metrics

| Metric | Description | Good/Bad Indicators |
|--------|-------------|---------------------|
| **TotPwr** | Total power consumption | Changes >1% warrant investigation |
| **DynPwr** | Dynamic power (switching) | Should correlate with activity changes |
| **LkgPwr** | Leakage power (static) | Changes >2% may indicate netlist/VT changes |
| **AQM** | Activity Quality Metric | >98% excellent, 93-98% acceptable, <90% poor |
| **GlobalClkPwr** | Clock tree power | High values may indicate clock gating opportunities |
| **SeqPwr** | Sequential element power | Correlates with flip-flop activity |
| **CombPwr** | Combinational logic power | Data path switching activity |

### VT Distribution Analysis

**ULVT (Ultra-Low VT)**:
- Fastest performance, highest leakage
- Target: Performance-critical paths only
- Regression indicator: >20% increase in ULVT cell count

**LVT (Low VT)**:
- Balanced speed/power trade-off
- Target: Majority of design (40-60%)
- Regression indicator: >30% decrease suggests over-optimization

**ULVTLL (Ultra-Low VT Low Leakage)**:
- Low leakage variant
- Target: Non-critical paths
- Regression indicator: Significant decrease may impact power

### Annotation Quality

| Metric | Description | Target |
|--------|-------------|--------|
| **AQM** | Overall Quality Metric | >98% excellent, 93-98% acceptable, <90% poor |
| **InpAnnotated** | Input pin annotation | >95% |
| **IntAnnotated** | Internal net annotation | >95% |
| **BBoxAnnotated** | Black box annotation | >95% |
| **SPCheckPass** | Static probability check | >95% |
| **AvgVal** | Overall validation score | >95% |

### Key Findings Categories

**Power Changes**:
- **<1%**: Normal variation, likely due to FSDB differences
- **1-5%**: Moderate change, review activity and configuration
- **>5%**: Significant change, requires investigation

**Annotation Changes**:
- **Δ<1%**: Negligible, acceptable variation
- **Δ1-5%**: Review annotation sources and mapping logs
- **Δ>5%**: Critical - investigate FSDB quality or mapping issues
- **Δ>50%**: Catastrophic - FSDB/annotation completely broken

## Validation Checklist

When reviewing comparison reports, verify:

- [ ] **Design Consistency**
  - Cell count matches between REF and EXP (or growth explained)
  - Sequential count unchanged (unless intentional)
  - Same design stepping/version

- [ ] **Configuration Matching**
  - Same voltage and temperature
  - Same test name
  - FSDB differences documented
  - PrimeTime versions compatible

- [ ] **Power Sanity**
  - Total power within expected range vs target
  - Dynamic/Leakage ratio reasonable (typically 60-80% dynamic)
  - No unexpected component power spikes

- [ ] **Annotation Quality**
  - AQM >90% (preferably >95%)
  - Input annotation >95%
  - Internal annotation >90%
  - No significant degradation vs REF (Δ<5%)

- [ ] **Activity Correlation**
  - Power changes correlate with activity changes
  - Clock activity matches expected frequency
  - Data activity reasonable for workload

- [ ] **VT Distribution**
  - ULVT usage reasonable (typically <50% of cells)
  - No unexpected ULVT explosion
  - Leakage proportional to ULVT usage

## Troubleshooting

### Issue: AQM Collapse (>50% drop)

**Symptoms**: AQM drops from >95% to <25%

**Root Causes**:
1. **FSDB Quality Issues**:
   - Incomplete waveform dump
   - Wrong time window
   - Missing hierarchy

2. **DOP Signal Mapping Failure**:
   - Design hierarchy changes
   - Signal name changes
   - Mapping file corruption

**Investigation Steps**:
```bash
# Check mapping reports
zcat partition.all_inclusive_mapping_summary.rpt.gz | head -50

# Verify FSDB time window
zcat partition.rollup.rpt.gz | grep -E "FSDB_(Start|End)"

# Check annotation sources
zcat partition.rollup.rpt.gz | grep -A10 "AQMskipped"
```

---

### Issue: Unexplained Leakage Regression

**Symptoms**: Leakage increases >20% without cell count growth

**Root Causes**:
1. **VT Shift**: ULVT usage increased
2. **Library Version**: Different characterization data
3. **Temperature**: Junction temperature changed

**Investigation Steps**:
```python
# Compare VT distributions
ref_cells = read_cell_power_csv("ref.compressed_cell_power.csv.gz")
exp_cells = read_cell_power_csv("exp.compressed_cell_power.csv.gz")

# Check ULVT usage
ref_ulvt_pct = (ref_cells['by_vt_class']['ch_ulvt_mac_3n']['count'] / 
                ref_cells['totals']['cell_count']) * 100
exp_ulvt_pct = (exp_cells['by_vt_class']['ch_ulvt_mac_3n']['count'] / 
                exp_cells['totals']['cell_count']) * 100
print(f"ULVT usage: {ref_ulvt_pct:.1f}% → {exp_ulvt_pct:.1f}%")
```

---

### Issue: Dynamic Power Reduction Despite Cell Growth

**Symptoms**: Dynamic power decreases while cell count increases >10%

**Root Cause**: **Almost always annotation quality collapse**

**Investigation**:
```python
# Check annotation quality
ref_metrics = read_rollup_report("ref.rollup.rpt.gz")
exp_metrics = read_rollup_report("exp.rollup.rpt.gz")

print(f"AQM: {ref_metrics['AQM']} → {exp_metrics['AQM']}")
print(f"Cells: {ref_metrics['CellCount']} → {exp_metrics['CellCount']}")
```

If AQM dropped >10%, the dynamic power reduction is an artifact - **fix annotation first**.

## Best Practices

1. **File Discovery (CRITICAL)**
   - ⭐ **ALWAYS start with `discover_reports_for_comparison`** to find all report paths upfront (PrimePower runs)
   - This eliminates repeated directory searches and path construction errors
   - Store the returned paths dictionary and reuse throughout analysis
   - Check the `files_found` dictionary to know which reports are available
   - ⚠️ **For Empower runs**, pass rollup and activity paths directly to individual tools — `discover_reports_for_comparison` assumes PrimePower directory layout

2. **Directory Organization**
   - Keep REF and EXP in separate directories
   - Use descriptive names (e.g., `baseline`, `optimization_v2`)
   - Archive reports with timestamps

3. **Comparison Workflow**
   - Always compare against a known-good baseline
   - Document expected changes (e.g., "optimization targeting clock tree")
   - Run validation checks before detailed analysis
   - Check AQM first - if <25%, results are unreliable

4. **Report Review Priority**
   1. Validation checklist (configuration, annotation quality)
   2. Key findings (>1% changes)
   3. Power breakdown (dynamic vs leakage)
   4. VT distribution analysis
   5. Cell family changes

5. **Root Cause Analysis**
   - Start with discover_reports_for_comparison to get all paths
   - Use comprehensive_power_comparison for high-level overview
   - Use VT analysis for leakage regressions
   - Check mapping reports for annotation issues
   - Compare variables reports for tool differences

6. **Documentation**
   - Save analysis reports with timestamps
   - Document root causes for power changes
   - Track improvements across design iterations
   - Keep configuration differences noted
